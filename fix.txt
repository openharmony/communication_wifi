1. sslUserDecision 会在同一次页面生命周期内复用到后续 SSL 错误，可能放大用户一次性确认的风险
位置：wifi/application/portal_login/entry/src/main/ets/pages/Index.ets

描述：

if (this.sslUserDecision === 1) {
  event.handler.handleConfirm();
  return;
} else if (this.sslUserDecision === 0) {
  event.handler.handleCancel();
  return;
}

用户一旦点击过“继续访问”或“取消”，后续新的 onSslErrorEvent 会直接复用上一次决定，而不再弹框确认。虽然这有助于消除重复弹窗，但也意味着：一次确认可能放行后续不同来源、不同证书问题的 SSL 错误事件。

影响：

如果同一页面生命周期内出现多个不同 SSL 异常，用户不会再次感知；
安全上会弱化“逐次确认”的保护效果；
一次“继续访问”可能被扩大应用到更多事件。
修复建议：

如果设计目标只是“合并同一波次的重复事件”，建议只对当前弹框期间缓存的 handler 复用结果，处理完成后立即恢复 sslUserDecision = -1；
不建议把一次选择扩展到弹框结束之后的新 SSL 事件。
例如：

this.sslErrorHandlers.forEach((handler) => {
  handler.handleConfirm();
});
this.sslErrorHandlers = [];
this.sslUserDecision = -1;
this.isSslDialogShowing = false;

2. sslErrorHandlers 无上限累积，异常风暴场景下可能短时堆积过多 handler
位置：同文件 onSslErrorEvent

描述：

this.sslErrorHandlers.push(event.handler);
if (this.isSslDialogShowing) {
  return;
}

在弹框展示期间，所有后续 SSL 错误都会把 handler 持续压入数组。如果底层频繁上报错误，数组可能短时间内快速膨胀。

影响：

极端场景下增加页面内存占用；
大量 handler 在用户点击后被批量执行，可能造成额外压力；
若某些 handler 已失效，批处理时可能带来不可预期行为。
修复建议：

增加上限保护或去重策略；
至少记录日志观测队列长度；
如框架允许，可只保留最新一个或按错误类型聚合。
3. 页面开始加载时仅在 !isSslDialogShowing 时重置状态，存在状态残留窗口
位置：onPageBegin

描述：

if (!this.isSslDialogShowing) {
  this.sslUserDecision = -1;
  this.sslErrorHandlers = [];
}

如果页面切换/重载时恰好仍处于弹框展示状态，则不会清空 sslUserDecision 和 sslErrorHandlers。这意味着旧页面上下文的 SSL 处理状态可能残留到新一轮页面事件中。

影响：

旧页面中的用户选择和 handler 队列可能影响新页面；
容易出现“上一轮状态泄漏到下一轮”的行为。
修复建议：

重新评估是否需要在页面重新开始加载时无条件清空状态；
如果不能无条件清空，至少要确保页面切换或销毁时统一重置。
